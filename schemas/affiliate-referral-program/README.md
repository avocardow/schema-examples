# Affiliate / Referral Program

## Overview

Affiliate and referral program management covering the full lifecycle from program creation to commission payouts. Supports multiple concurrent programs with independent commission rules, cookie-based and server-side attribution, and a complete tracking funnel from clicks through conversions to commissions. Designed from a study of 8 systems: Rewardful, Tapfiliate, Post Affiliate Pro, Impact.com, GrowSurf, Referral Rock, LeadDyno, and Refferq.

Key design decisions: programs as top-level containers for commission configuration (supports multiple concurrent programs with different rules); affiliates as separate entities from users (not all users are affiliates, and affiliates carry program-specific data like commission rates and payout info); full click → referral → conversion funnel tracking (append-only clicks for analytics, referrals for attribution, conversions for completed actions); commissions as independent entities with their own lifecycle (pending → approved → paid, decoupled from conversions to support recurring commissions); batched payouts with line-item detail (reduces transfer fees, matches industry patterns); balance + ledger model for financial tracking (every money movement is an append-only ledger entry, with running balance for quick queries).

## Schema

The pseudo code below is the **single source of truth** for all format implementations. See [CONTRIBUTING.md](../../CONTRIBUTING.md) for pseudo code conventions.

### 1. programs

Top-level container for affiliate/referral program configuration. Each program defines its own commission rules, cookie duration, and attribution model. Supports multiple concurrent programs (e.g., a referral program for customers and a separate affiliate program for content creators). Inspired by Rewardful Campaigns, Tapfiliate Programs, and Post Affiliate Pro Campaigns.

```pseudo
table programs {
  id                uuid primary_key default auto_generate
  name              string not_null              -- Program name (e.g., "Customer Referral Program", "Partner Affiliate Program").
  slug              string unique not_null       -- URL-friendly identifier for the program.
  description       string nullable              -- Description of the program terms and benefits.
  status            enum(draft, active, paused, archived) not_null default draft
                                                 -- draft: program being configured.
                                                 -- active: accepting affiliates and tracking conversions.
                                                 -- paused: temporarily stopped (no new tracking).
                                                 -- archived: permanently closed.
  commission_type   enum(percentage, flat, hybrid) not_null default percentage
                                                 -- percentage: percentage of conversion amount.
                                                 -- flat: fixed amount per conversion.
                                                 -- hybrid: percentage + flat fee.
  commission_percentage decimal nullable         -- Default commission percentage (0.00–1.00). Required when commission_type = percentage or hybrid.
  commission_flat   integer nullable             -- Default flat commission in smallest currency unit. Required when commission_type = flat or hybrid.
  currency          string not_null              -- ISO 4217 currency code for commissions.
  cookie_duration   integer not_null default 30  -- Attribution cookie lifetime in days.
  attribution_model enum(first_touch, last_touch) not_null default last_touch
                                                 -- first_touch: first affiliate to refer gets credit.
                                                 -- last_touch: most recent affiliate gets credit.
  min_payout        integer not_null default 0   -- Minimum balance required for payout in smallest currency unit.
  auto_approve      boolean not_null default false -- Whether commissions are auto-approved or require manual review.
  is_public         boolean not_null default true  -- Whether the program is publicly listed for affiliate signup.
  terms_url         string nullable              -- URL to program terms and conditions.
  created_by        uuid not_null references users(id) on_delete restrict  -- Admin who created the program.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(status)
    index(created_by)
  }
}
```

**Design notes:**
- `commission_type` + `commission_percentage` + `commission_flat` define the default commission structure. Affiliates can override via `affiliates.custom_commission_rate`.
- `cookie_duration` determines how long a click is attributed to an affiliate (e.g., 30 days = a conversion within 30 days of the click credits the affiliate).
- `attribution_model` determines which affiliate gets credit when a visitor clicks multiple affiliate links.
- `auto_approve` controls whether commissions go straight to "approved" or require manual review.

### 2. affiliates

Individual affiliate accounts representing a user's enrollment in a specific program. An affiliate has program-specific settings like custom commission rates and payout information. A user can be an affiliate in multiple programs. Inspired by Rewardful Affiliates, Tapfiliate Affiliates, and LeadDyno Affiliates.

```pseudo
table affiliates {
  id                uuid primary_key default auto_generate
  program_id        uuid not_null references programs(id) on_delete cascade
                                                 -- Which program this affiliate belongs to.
                                                 -- Cascade: removing a program removes all its affiliates.
  user_id           uuid not_null references users(id) on_delete cascade
                                                 -- The user account. Cascade: removing a user removes their affiliate accounts.
  referral_code     string unique not_null       -- Unique referral code for this affiliate (e.g., "JOHN2024").
  coupon_code       string nullable              -- Optional coupon code for tracking conversions without links.
  status            enum(pending, active, suspended, rejected) not_null default pending
                                                 -- pending: application submitted, awaiting approval.
                                                 -- active: approved and can earn commissions.
                                                 -- suspended: temporarily disabled (policy violation).
                                                 -- rejected: application denied.
  custom_commission_rate decimal nullable         -- Affiliate-specific commission override (0.00–1.00). Null = use program default.
  payout_method     string nullable              -- Preferred payout method (e.g., "paypal", "bank_transfer", "stripe").
  payout_email      string nullable              -- Email for PayPal/payment provider payouts.
  metadata          json nullable default {}     -- Extensible data (social profiles, website URL, traffic sources, etc.).
  referred_by       uuid nullable references affiliates(id) on_delete set_null
                                                 -- The affiliate who recruited this affiliate (for referral chains).
                                                 -- Set null: original referrer can be removed without affecting this affiliate.
  approved_at       timestamp nullable           -- When the affiliate was approved.
  suspended_at      timestamp nullable           -- When the affiliate was suspended.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    composite_unique(program_id, user_id)        -- A user can only be an affiliate in a program once.
    index(user_id)
    index(status)
    index(referred_by)
  }
}
```

**Design notes:**
- `composite_unique(program_id, user_id)` prevents duplicate enrollment — one affiliate account per user per program.
- `referral_code` is globally unique across all programs (simpler routing, avoids collisions).
- `coupon_code` enables tracking via discount codes (e.g., "Use code JOHN20 for 20% off"). Null = no coupon assigned.
- `referred_by` supports affiliate-recruits-affiliate chains without full multi-tier commission tables.
- `custom_commission_rate` overrides the program's default rate for high-performing affiliates.

### 3. affiliate_links

Tracking links generated by affiliates for specific destination URLs. Each link has a unique token embedded in the URL for click tracking. Affiliates can create multiple links for different landing pages or campaigns. Inspired by Rewardful Affiliate Links and Tapfiliate tracking URLs.

```pseudo
table affiliate_links {
  id                uuid primary_key default auto_generate
  affiliate_id      uuid not_null references affiliates(id) on_delete cascade
                                                 -- Which affiliate owns this link.
                                                 -- Cascade: removing an affiliate removes all their links.
  token             string unique not_null       -- Unique token for URL tracking (e.g., "a1b2c3d4").
  slug              string nullable              -- Optional vanity slug (e.g., "johns-deal"). Null = use token.
  destination_url   string not_null              -- Target URL the link redirects to.
  label             string nullable              -- Internal label for the affiliate's reference (e.g., "Blog Header Link").
  total_clicks      integer not_null default 0   -- Denormalized click count for quick display.
  total_conversions integer not_null default 0   -- Denormalized conversion count for quick display.
  is_active         boolean not_null default true -- Whether this link is currently tracking.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(affiliate_id)
  }
}
```

**Design notes:**
- `token` is the tracking identifier embedded in URLs (e.g., `https://example.com/?ref=a1b2c3d4`).
- `slug` enables vanity URLs (e.g., `https://example.com/?ref=johns-deal`).
- `total_clicks` and `total_conversions` are denormalized counters updated on each click/conversion for dashboard display without expensive aggregation queries.

### 4. clicks

Raw click events on affiliate links. Append-only, high-volume table for tracking every click with device/browser metadata. Used for analytics and attribution. Each click generates a unique `click_id` stored in the visitor's cookie for server-side attribution. Inspired by Refferq click tracking and Post Affiliate Pro click logs.

```pseudo
table clicks {
  id                uuid primary_key default auto_generate
  affiliate_link_id uuid not_null references affiliate_links(id) on_delete cascade
                                                 -- Which link was clicked.
                                                 -- Cascade: removing a link removes its click history.
  click_id          string unique not_null       -- Unique click identifier stored in cookie for attribution.
  ip_address        string nullable              -- Visitor's IP address (hashed or anonymized for privacy).
  user_agent        string nullable              -- Browser user agent string.
  referer_url       string nullable              -- HTTP referer header (where the click came from).
  landing_url       string nullable              -- Actual URL the visitor landed on.
  country           string nullable              -- Geo-resolved country code (ISO 3166-1 alpha-2).
  device_type       string nullable              -- Device category (e.g., "desktop", "mobile", "tablet").
  is_unique         boolean not_null default true -- Whether this is the first click from this visitor (based on IP/cookie).
  created_at        timestamp default now

  indexes {
    index(affiliate_link_id, created_at)
    index(created_at)
  }
}
```

**Design notes:**
- **Append-only** — clicks are never updated or deleted. They form the raw analytics trail.
- `click_id` is stored in the visitor's first-party cookie and used to attribute conversions back to this click.
- `is_unique` distinguishes first-time visitors from repeat clicks for accurate analytics.
- `ip_address` should be hashed or anonymized in production for GDPR/privacy compliance.
- Security: raw click data is sensitive — restrict access to program admins only.

### 5. referrals

Attributed visits/leads from affiliate clicks. A referral is created when a click is successfully attributed to an affiliate and the visitor takes a meaningful action (e.g., visits a landing page, signs up). Tracks the visitor through the funnel from visit to lead to conversion. Inspired by Rewardful Referrals (Visitor → Lead → Conversion) and LeadDyno's funnel model.

```pseudo
table referrals {
  id                uuid primary_key default auto_generate
  affiliate_id      uuid not_null references affiliates(id) on_delete cascade
                                                 -- Which affiliate is credited with this referral.
                                                 -- Cascade: removing an affiliate removes their referrals.
  click_id          uuid nullable references clicks(id) on_delete set_null
                                                 -- The click that initiated this referral.
                                                 -- Set null: click history can be purged without losing referrals.
  visitor_id        string nullable              -- Anonymous visitor identifier (cookie/device ID).
  email             string nullable              -- Visitor's email if captured (lead stage).
  status            enum(visit, lead, converted, expired) not_null default visit
                                                 -- visit: visitor landed but hasn't identified.
                                                 -- lead: visitor provided contact info (email, signup).
                                                 -- converted: visitor completed a conversion (purchase, subscription).
                                                 -- expired: attribution window expired without conversion.
  landing_url       string nullable              -- URL the visitor landed on.
  metadata          json nullable                -- Additional tracking data (UTM params, campaign tags, etc.).
  converted_at      timestamp nullable           -- When the referral converted. Null = not yet converted.
  expires_at        timestamp nullable           -- When the attribution window expires (based on cookie_duration).
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(affiliate_id, status)
    index(status)
    index(email)
    index(visitor_id)
  }
}
```

**Design notes:**
- Referrals represent the attributed visitor journey: visit → lead → converted.
- `visitor_id` enables cookieless server-side tracking (stored server-side, not in browser).
- `expires_at` is calculated from the program's `cookie_duration` at referral creation time.
- `email` is captured when the visitor identifies themselves (e.g., newsletter signup, account creation).

### 6. conversions

Completed conversion events that generate commissions. A conversion represents a valuable action: a purchase, subscription signup, trial activation, or any custom event defined by the program. Uses polymorphic references to link to external entities (orders, subscriptions) without creating a hard dependency. Inspired by Tapfiliate Conversions and Rewardful Sales.

```pseudo
table conversions {
  id                uuid primary_key default auto_generate
  referral_id       uuid not_null references referrals(id) on_delete restrict
                                                 -- Which referral led to this conversion.
                                                 -- Restrict: don't delete referrals with conversions.
  affiliate_id      uuid not_null references affiliates(id) on_delete restrict
                                                 -- Which affiliate earned this conversion.
                                                 -- Restrict: don't delete affiliates with conversions.
  external_id       string nullable              -- ID in the external system (order ID, subscription ID, etc.).
  reference_type    string nullable              -- Type of external entity (e.g., "order", "subscription", "signup").
  amount            integer not_null default 0   -- Conversion value in smallest currency unit.
  currency          string not_null              -- ISO 4217 currency code.
  status            enum(pending, approved, rejected, reversed) not_null default pending
                                                 -- pending: conversion recorded, awaiting validation.
                                                 -- approved: conversion validated, commission can be calculated.
                                                 -- rejected: conversion invalidated (fraud, return, etc.).
                                                 -- reversed: previously approved conversion reversed (chargeback, refund).
  metadata          json nullable                -- Additional conversion data (product info, plan details, etc.).
  approved_at       timestamp nullable           -- When the conversion was approved.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(referral_id)
    index(affiliate_id, status)
    index(external_id)
    index(status)
    index(created_at)
  }
}
```

**Design notes:**
- `external_id` + `reference_type` provide a polymorphic link to external systems (orders, subscriptions, etc.) without creating a hard FK dependency on e-commerce or subscription domains.
- `amount` is the total conversion value used for commission calculation (e.g., order total, subscription amount).
- Conversion `status` is separate from commission `status` — a conversion can be approved while its commission is still pending payout.

### 7. commissions

Individual commission records generated from conversions. Each commission has its own lifecycle: pending → approved → paid (or voided). Commissions are the unit of financial tracking — they are included in payouts and recorded in the balance ledger. Inspired by Rewardful Commissions, Tapfiliate Commissions, and Post Affiliate Pro commission management.

```pseudo
table commissions {
  id                uuid primary_key default auto_generate
  conversion_id     uuid not_null references conversions(id) on_delete restrict
                                                 -- Which conversion generated this commission.
                                                 -- Restrict: don't delete conversions with commissions.
  affiliate_id      uuid not_null references affiliates(id) on_delete restrict
                                                 -- Which affiliate earns this commission.
                                                 -- Restrict: don't delete affiliates with commissions.
  program_id        uuid not_null references programs(id) on_delete restrict
                                                 -- Which program this commission belongs to.
                                                 -- Restrict: don't delete programs with commissions.
  amount            integer not_null             -- Commission amount in smallest currency unit.
  currency          string not_null              -- ISO 4217 currency code.
  status            enum(pending, approved, paid, voided) not_null default pending
                                                 -- pending: awaiting review/approval.
                                                 -- approved: approved and available for payout (added to affiliate balance).
                                                 -- paid: included in a completed payout.
                                                 -- voided: canceled (fraud, conversion reversed, etc.).
  commission_type   enum(percentage, flat, hybrid) not_null
                                                 -- The type used to calculate this commission (snapshot from program/affiliate).
  commission_rate   decimal nullable             -- The percentage rate used (snapshot). Null when flat only.
  commission_flat   integer nullable             -- The flat amount used (snapshot). Null when percentage only.
  tier_level        integer not_null default 1   -- Commission tier (1 = direct, 2+ = sub-affiliate tiers).
  approved_at       timestamp nullable           -- When the commission was approved.
  paid_at           timestamp nullable           -- When the commission was paid out.
  voided_at         timestamp nullable           -- When the commission was voided.
  voided_reason     string nullable              -- Why the commission was voided.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(conversion_id)
    index(affiliate_id, status)
    index(program_id, status)
    index(status)
    index(created_at)
  }
}
```

**Design notes:**
- `commission_type`, `commission_rate`, `commission_flat` are snapshots of the rates at the time the commission was calculated. This preserves the historical record even if program rates change later.
- `tier_level` supports multi-tier commissions: level 1 = direct sale, level 2+ = commissions earned from sub-affiliate sales. Multi-tier logic lives in application code.
- Commission lifecycle: pending → approved (manual or auto) → paid (via payout) or voided (fraud/reversal).

### 8. payouts

Batched payout disbursements to affiliates. Payouts aggregate multiple approved commissions into a single payment. Supports configurable payout methods and provider tracking. Inspired by Rewardful Payouts and the marketplace domain's payout pattern.

```pseudo
table payouts {
  id                uuid primary_key default auto_generate
  affiliate_id      uuid not_null references affiliates(id) on_delete restrict
                                                 -- Which affiliate is receiving this payout.
                                                 -- Restrict: don't delete affiliates with payout records.
  payout_number     string unique not_null       -- Human-readable payout reference (e.g., "PAY-20240315-X1Y2").
  status            enum(pending, processing, completed, failed, canceled) not_null default pending
                                                 -- pending: scheduled but not yet initiated.
                                                 -- processing: payout initiated with payment provider.
                                                 -- completed: funds transferred successfully.
                                                 -- failed: transfer failed (retry or investigate).
                                                 -- canceled: payout canceled before processing.
  currency          string not_null              -- ISO 4217 currency code.
  amount            integer not_null             -- Payout amount in smallest currency unit.
  fee               integer not_null default 0   -- Transfer fee deducted.
  net_amount        integer not_null             -- Amount affiliate receives: amount - fee.
  payout_method     string nullable              -- Payment method used (e.g., "paypal", "bank_transfer", "stripe").
  provider_id       string nullable              -- Provider-side payout/transfer ID for reconciliation.
  note              string nullable              -- Internal note about the payout.
  completed_at      timestamp nullable           -- When the payout was completed.
  created_at        timestamp default now
  updated_at        timestamp default now on_update

  indexes {
    index(affiliate_id, status)
    index(status)
    index(created_at)
  }
}
```

**Design notes:**
- Payouts batch multiple approved commissions into a single disbursement.
- `net_amount` = `amount` - `fee`. The affiliate sees the net amount deposited.
- `provider_id` enables reconciliation with external payment providers (Stripe, PayPal, etc.).

### 9. payout_items

Individual commission items within a payout. Links each commission to its parent payout for detailed reconciliation and transparency. Inspired by marketplace payout_items pattern.

```pseudo
table payout_items {
  id                uuid primary_key default auto_generate
  payout_id         uuid not_null references payouts(id) on_delete cascade
                                                 -- Parent payout. Cascade: removing a payout removes its items.
  commission_id     uuid not_null references commissions(id) on_delete restrict
                                                 -- Which commission is included in this payout.
                                                 -- Restrict: don't delete commissions included in payouts.
  amount            integer not_null             -- Commission amount included in this payout.
  created_at        timestamp default now

  indexes {
    composite_unique(payout_id, commission_id)   -- Each commission appears once per payout.
    -- composite_unique(payout_id, commission_id) covers index(payout_id) via leading column.
  }
}
```

### 10. affiliate_balances

Running balance per affiliate. Updated transactionally with every balance-changing event (commission approved, payout completed, voided, adjustment). Provides quick-access current balance without summing the entire ledger. Inspired by marketplace vendor_balances and Stripe Connect balance.

```pseudo
table affiliate_balances {
  id                uuid primary_key default auto_generate
  affiliate_id      uuid unique not_null references affiliates(id) on_delete cascade
                                                 -- One balance per affiliate.
                                                 -- Cascade: deleting an affiliate removes its balance.
  currency          string not_null              -- ISO 4217 currency code.
  available         integer not_null default 0   -- Funds available for payout (approved commissions minus pending payouts).
  pending           integer not_null default 0   -- Commissions earned but not yet approved.
  total_earned      integer not_null default 0   -- Lifetime total commissions earned.
  total_paid_out    integer not_null default 0   -- Lifetime total payouts completed.
  updated_at        timestamp default now on_update
}
```

**Design notes:**
- One-to-one with `affiliates` (enforced by `unique` on `affiliate_id`).
- `available` = funds the affiliate can withdraw. `pending` = commissions awaiting approval.
- `total_earned` and `total_paid_out` provide lifetime summaries without querying the ledger.
- Balance must always be updated transactionally alongside `balance_transactions` entries.

### 11. balance_transactions

Append-only ledger of all balance changes for affiliates. Every credit or debit to an affiliate's balance is recorded as a transaction entry. Provides a complete audit trail for financial reconciliation. Inspired by marketplace balance_transactions and Stripe Balance Transactions.

```pseudo
table balance_transactions {
  id                uuid primary_key default auto_generate
  affiliate_id      uuid not_null references affiliates(id) on_delete restrict
                                                 -- Which affiliate's balance this transaction affects.
                                                 -- Restrict: don't delete affiliates with transaction history.
  type              enum(commission, payout, reversal, adjustment) not_null
                                                 -- commission: commission approved (credit).
                                                 -- payout: funds paid out to affiliate (debit).
                                                 -- reversal: commission voided or conversion reversed (debit).
                                                 -- adjustment: manual adjustment by admin (credit or debit).
  amount            integer not_null             -- Transaction amount in smallest currency unit. Positive = credit, negative = debit.
  currency          string not_null              -- ISO 4217 currency code.
  running_balance   integer not_null             -- Affiliate's available balance after this transaction.
  reference_type    string nullable              -- What entity this transaction references (e.g., "commission", "payout").
  reference_id      string nullable              -- ID of the referenced entity.
  description       string nullable              -- Human-readable description of the transaction.
  created_at        timestamp default now

  indexes {
    index(affiliate_id, created_at)
    index(type)
    index(reference_type, reference_id)
  }
}
```

**Design notes:**
- **Append-only** — transactions are never updated or deleted. Corrections are made by adding new adjustment entries.
- `running_balance` captures the balance snapshot at each point, enabling reconciliation without replaying the entire ledger.
- `reference_type` + `reference_id` provide a polymorphic link to the source event (commission, payout, etc.).

## Relationships

### One-to-Many

```
users                  1 ──── * affiliates               (user → affiliate accounts)
programs               1 ──── * affiliates               (program → enrolled affiliates)
affiliates             1 ──── * affiliate_links           (affiliate → tracking links)
affiliate_links        1 ──── * clicks                    (link → click events)
affiliates             1 ──── * referrals                 (affiliate → attributed referrals)
referrals              1 ──── * conversions               (referral → conversion events)
affiliates             1 ──── * conversions               (affiliate → earned conversions)
conversions            1 ──── * commissions               (conversion → commission records)
affiliates             1 ──── * commissions               (affiliate → earned commissions)
programs               1 ──── * commissions               (program → generated commissions)
affiliates             1 ──── * payouts                   (affiliate → payout disbursements)
payouts                1 ──── * payout_items              (payout → line items)
affiliates             1 ──── 1 affiliate_balances        (affiliate → balance)
affiliates             1 ──── * balance_transactions      (affiliate → ledger entries)
affiliates             1 ──── * affiliates                (affiliate → recruited affiliates, self-ref via referred_by)
```

### Many-to-Many (via junction tables)

```
payouts ←── payout_items ──→ commissions                 (payouts × commissions)
```

## Best Practices

- **Commission calculation**: When a conversion is approved, calculate the commission using the affiliate's custom rate (if set), otherwise fall back to the program's default rate. Snapshot the rate on the commission record.
- **Attribution flow**: Click → set cookie with click_id → create referral with expires_at → on conversion, find referral by cookie/visitor_id → create conversion → calculate commission.
- **Balance integrity**: Always update `affiliate_balances` and insert `balance_transactions` in the same database transaction. The ledger is the source of truth — if balance and ledger disagree, the ledger wins.
- **Payout workflow**: Batch approved commissions into a payout → mark commissions as "paid" → debit affiliate balance → record balance transaction → initiate payment provider transfer.
- **Fraud prevention**: Use the pending → approved commission workflow to review suspicious conversions before crediting affiliates. Monitor for click fraud (high click-to-conversion ratios, bot traffic).
- **Cookie expiry**: Set `referrals.expires_at` based on `programs.cookie_duration` at referral creation time. Expired referrals should not generate commissions.
- **Conversion reversal**: When a conversion is reversed (refund, chargeback), void the commission and add a reversal balance transaction. Never delete the original records.
- **Soft deletion**: Affiliates with commissions or payouts should never be hard-deleted. Use `status = suspended` or `rejected` instead. Their financial history must be preserved.
- **Access control**: Use Auth / RBAC for permission checks. Affiliates see only their own data (links, clicks, referrals, commissions, payouts). Program admins see all data within their programs.
